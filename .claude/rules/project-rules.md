# 项目运营规则

## 核心原则
1. **最小改动**：只修改需求指定的内容
2. **尊重现有规范**：遵循项目已有的代码风格与模式
3. **渐进式更改**：优先分阶段迭代，禁止一次性大规模改动

## 修改前校验原则
- 修改文件前必须先完整阅读文件内容（先阅读 → 再编辑）
- 修改前先检索项目中是否已有相似实现代码
- 修改完成后，检查是否对关联文件造成影响

## 项目结构维护
- 严格沿用现有文件夹结构，禁止随意新建文件夹
- 新增代码必须和原有代码风格保持一致
- 新增第三方库 / 包之前，先确认能否用现有依赖实现需求

## 文件大小限制
- 单个文件：超过 300 行需考虑拆分
- 单个函数：超过 50 行需考虑拆分
- 单个组件：超过 200 行需拆分为子组件

## 杜绝代码重复（DRY 原则）
- 相同逻辑重复出现 2 次及以上，必须抽离为公共函数 / 公共组件
- 相同常量值（颜色、接口地址等）分散在多文件时，统一集中在一处定义后全局引用

## 先定义用户场景
开发实现前先梳理用户使用场景。
- 分为四类场景梳理：正常流程 / 异常处理 / 边界值 / 故障恢复
- 示例：登录 → 登录成功 / 密码错误 / 空输入提交 / 网络断开
- 场景定义完成后，再开始编码实现

## TDD（测试驱动开发）
遵循「先定义失败条件 → 最简代码通过测试 → 重构优化结构」三步循环：
1. **Red（红）**：先编写会执行失败的测试用例
2. **Green（绿）**：编写最少代码让测试通过
3. **Refactor（重构）**：在保持测试通过的前提下优化代码结构

- 测试失败时只能修改业务代码，**禁止通过修改测试代码强行通过**
- 测试隔离原则：每条测试用例必须可独立运行
- 命名规范：采用 `describe("功能名称")` + `it("当……则……")` 固定格式

## 错误处理规范
- 禁止空 catch 代码块，不允许静默忽略异常
- 调试用 `console.log` 调试结束后必须全部删除
- 错误提示信息中禁止暴露项目内部敏感信息
- 发生错误时，同一方案重试不超过 3 次
- 发现陌生文件 / 未知配置时，禁止直接删除，先确认用途
- 必须理解错误根源后再修改；无法定位原因时主动提问

## 安全审查
新增 API 接口或处理用户输入时，必须逐项检查：
- 认证校验：接口是否已配置身份认证中间件
- 权限鉴权：是否校验资源归属，防止越权访问他人数据
- 输入校验：服务端是否校验参数类型、范围、格式合法性
- 输出转义：用户输入是否会直接嵌入 HTML / SQL 语句中

## 事前风险排查（Pre-mortem）
任务收尾前提前排查所有潜在失败场景：
- 空值 / 未定义、空数组、海量数据、网络超时、并发访问
- 确认除正常流程外，所有边界异常场景均已处理
- 排查发现的问题，必须在任务结束前全部修复

## Git 提交规范
- 仅在用户明确要求时执行提交操作
- 提交说明以**变更原因**为核心进行描述
